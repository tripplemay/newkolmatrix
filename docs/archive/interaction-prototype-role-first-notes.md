# KOLMatrix Role-first 工作台方案

> 对比原型：`docs/product/interaction-prototype.html`
>
> 新版原型：`docs/product/interaction-prototype-role-first.html`
>
> 目标：验证“角色驾驶舱 → 单项目工作 → 跨角色交接”是否比“跨项目环节工作台”更符合真实协作心智。

## 1. 本方案不沿用的前提

新版没有把 Brief、Match、Reach、Delivery 当作四个全员共享的一级工作台，也没有把六环节继续做成所有角色共用的页面模板。

这六个概念仍然存在，但只作为项目状态与交接依据。用户执行工作时，看到的是自己的职责对象：负责人看到决策，合作经理看到创作者合作管线，商务运营看到商业控制条件。

## 2. 新的信息架构

```text
角色驾驶舱
├── 现在该做：只包含当前角色拥有的决定或执行
├── 项目责任视图：显示每个项目中“我的下一动作”
└── 协作交接：发送人 → 接收人 → 完成条件 → 可退回原因
         │
         ▼
单项目空间
├── 共享项目事实：目标、预算、市场、当前责任人、健康度
├── 当前角色专属工作台
├── 项目协作责任图
└── 当前项目、当前角色的数据边界 Agent
```

不存在跨项目 Match、Reach、Delivery 执行页。跨项目页面只用于分流和发现，真正操作必须回到单项目上下文。

## 3. 三角色定义

| 角色 | 对什么结果负责 | 核心工作对象 | 拥有的闸门 | 明确不负责 |
|---|---|---|---|---|
| 营销负责人 | 项目增长结果、预算配置和关键取舍 | 项目目标、创作者组合、预算例外、复盘结论 | 批准组合、批准预算例外、采纳复盘 | 不发信、不谈判、不签合同、不放款 |
| 创作者合作经理 | 创作者供给、沟通谈判、交付质量和长期关系 | 候选池、对话、报价、内容版本、关系档案 | 发信、日常授权内谈判、内容审核与发布交接 | 不批准预算例外、不签合同、不操作资金 |
| 商务运营 | 合同、合规、托管、放款条件和审计完整性 | 合同资料、税务/主体、披露证据、托管与款项 | 合同与合规确认、逐笔放款 | 不选人、不判断内容创意、不读取私人联系方式 |

### 为什么把“财务”改为“商务运营”

单纯财务角色不足以承接合同主体、税务资料、市场披露、托管条件和放款证据之间的连续责任。新版将其定义为商务控制角色，但仍保留严格数据边界，不参与创作者选择与内容判断。

## 4. 三套不同的项目工作台

### 4.1 营销负责人：项目决策室

核心问题不是“这个环节有什么数据”，而是：

- 项目是否仍然朝目标推进；
- 哪个取舍必须由我承担；
- 批准后责任交给谁；
- 哪些执行不应该落到我身上。

因此页面由目标健康度、组合方案对比、预算例外和项目复盘构成。批准组合只让方案生效并交给合作经理，不会同时发送邮件。

### 4.2 创作者合作经理：创作者合作台

核心对象是连续的创作者关系，而不是四个分散模块：

```text
候选 → 人工审阅 → 组合生效 → 触达 → 回复/谈判 → 内容交付 → 关系沉淀
```

候选管线、沟通收件箱、内容交付和关系档案使用不同的信息密度与操作方式。超过授权额度的报价会生成一张交接单，不会把营销负责人拉进整个沟通工作台。

### 4.3 商务运营：商业控制台

商务运营使用精确台账和条件清单，而不是生成式推荐卡：

- 每位创作者独立核对合同、主体、税务、披露和托管；
- 缺口必须退回明确责任人；
- 放款逐笔执行，且必须消费合作经理提交的交付证据；
- 不满足条件时只展示缺什么，不提供绕过入口。

## 5. 跨角色协作契约

每次交接必须包含：

```text
发送角色
接收角色
项目 ID
业务对象 ID
交付物或决定
完成条件
承诺时间
证据
状态：待接收 / 处理中 / 待决定 / 已完成 / 已退回
退回原因
```

原型中的主链为：

```text
营销负责人确认目标与预算
  → 创作者合作经理接收发现任务
  → 合作经理提交 14 人组合
  → 营销负责人批准组合
  → 合作经理触达、谈判和审核交付
  → 商务运营补齐合同、合规与托管
  → 合作经理提交交付证据
  → 商务运营逐笔放款
  → 营销负责人采纳复盘
```

阶段没有消失，但阶段的意义变成“哪个责任对象已完成并可以交接”，而不是一个所有人都可以进入的进度页面。

## 6. Agent 在新版中的位置

Agent 仍然是高频入口，但不再拥有一套模糊的全局能力。

| Agent | 主要产出 | 数据边界 | 不执行 |
|---|---|---|---|
| Campaign Director | 目标摘要、组合对比、预算建议、复盘草案 | 全项目目标、预算与结果 | 不联系创作者、不放款 |
| Creator Partner Agent | 候选重排、逐人邀约、回复草稿、交付检查建议 | 当前合作经理负责的项目和创作者 | 不批准预算例外、不作未确认承诺 |
| Commercial Control Agent | 资料缺口、合规校验、放款准备度、审计查询 | 合同、合规和交易字段 | 不读取联系方式、不自动放款 |

切换角色时会重建对话上下文。进入项目后，Agent 默认只在当前项目中工作。

## 7. 与现有原型的结构对比

| 设计问题 | 现有原型 | Role-first 新版 |
|---|---|---|
| 工作起点 | 共享驾驶舱，待办与项目环节混合 | 每个角色独立驾驶舱，只出现本人责任 |
| 一级导航 | Brief / Match / Reach / Delivery 等业务环节 | 今日驾驶舱 / 项目 / 当前角色资产与控制面 |
| 执行作用域 | 环节页跨项目切换 | 先选项目，再进入角色专属工作台 |
| 项目阶段 | 共享的六环节页面和进度条 | 共享状态事实 + 责任交接，不作为统一工作台 |
| 角色差异 | 同一页面按权限隐藏按钮 | 页面结构、信息密度、对象和动作均不同 |
| 营销负责人 | 能进入多数执行页面 | 只看目标、决策、例外和复盘 |
| 合作经理 | 在多个环节页间切换 | 一条连续创作者合作管线 |
| 财务/商务 | 在项目历史环节中找结算 | 独立商业控制台和逐笔条件清单 |
| 协作 | 审批单附着在公共工作流上 | 交接单是角色之间的正式业务对象 |
| Agent 上下文 | 角色 + 当前模块，容易跨项目 | 角色 + 单项目 + 职责对象三重约束 |
| AI-native 表达 | 各模块都有相似 Agent 卡片 | Agent 为不同角色生成不同类型的工作成果 |

## 8. 建议比对路径

### 路径 A：角色是否清楚

1. 打开新版，依次切换营销负责人、创作者合作经理、商务运营。
2. 比较三个驾驶舱的“责任声明”“现在该做”和“职责边界”。
3. 判断是否能在 10 秒内回答：我对什么结果负责、现在做什么、哪些事不归我。

### 路径 B：单项目心流

1. 以营销负责人打开“原神 · 东南亚女性向 Q2”。
2. 进入组合决策，批准 $9,800 方案。
3. 切换合作经理，仍进入同一项目，查看已生效组合和沟通谈判。
4. 观察执行期间是否需要进入任何跨项目环节页。

### 路径 C：完整交接与资金动作

1. 合作经理在内容交付中通过 AyuGaming 终稿并交接证据。
2. 切换商务运营，进入同一项目的“托管与放款”。
3. 确认 $1,800 放款。
4. 查看项目协作时间线和审计记录是否能解释责任如何转移。

## 9. 本轮需要用户判断的产品决策

1. 是否认可“六环节保留为项目状态，但不再作为一级工作台”？
2. 是否认可将财务扩展为“商务运营”，承接合同、合规、托管和放款的完整责任？
3. 合作经理是否应同时拥有内容交付审核，还是需要独立的内容运营角色？
4. 营销负责人是否应该批准所有组合，还是只审批超过预算或风险阈值的组合？
5. 跨项目收件箱是否只做分流，并强制回到项目内执行？

这些问题决定下一版的角色模型和业务实体，不应继续通过隐藏按钮或改导航名称来回避。
