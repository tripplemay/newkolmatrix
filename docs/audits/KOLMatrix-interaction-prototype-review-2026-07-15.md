# KOLMatrix AI-native 交互原型全面审查报告

> 审查日期：2026-07-15  
> 审查对象：`docs/product/interaction-prototype.html`  
> 结论：**产品方向基本成立，但原型尚不具备定稿条件，不建议按当前版本直接进入全面开发。**

## 0. 执行摘要

当前原型已经完成了重构中最重要的一次方向转换：它不再把 AI 放在传统 SaaS 页面角落，而是以常驻 Agent 对话、生成式画布、组合级决策、可解释匹配和人类闸门作为主轴。就“是否摆脱旧 kolmatrix 的菜单 + 表格 + 手工状态维护”而言，答案是 **基本摆脱了**。

但“看起来像 AI-native”与“工作流已经可作为开发蓝本”是两件事。深查状态机、权限、审批和跨模块状态后，发现多个核心机制仍是展示性闭环：阶段可以绕过前置条件，审批结果没有绑定业务对象，报价与批量发信阈值没有执行，角色数据边界只做到模块级，默认 BD 的首屏建项目入口还会被权限关键词误拦截。

因此本次结论分成两层：

- **重构方向：通过。** Agent 主轴、决策者定位、游戏垂类工作流和可信数据表达已经形成明确产品个性。
- **原型定稿：不通过。** 当前版本不足以作为无歧义开发规格；若直接开发，会把权限漏洞、假审批和不完整状态机一起固化。

建议状态：**继续原型迭代，完成 P0/P1 后重新验收，再冻结产品形态。**

## 3. 审查方法与限制

### 审查方式

1. 通读原型的数据模型、角色模型、六阶段模型、渲染器、Agent 意图分发、事件委托和所有闸门逻辑。
2. 逐项对照 `ai-native-usage.md` 中的总纲、六个业务模块、角色数据边界和三项量化阈值。
3. 从 `lead`、`bd`、`finance` 三类角色反向检查可见数据、可执行动作和跨角色交接。
4. 检查桌面/移动端 CSS、空态、加载态、失败态、撤销与不可逆动作。
5. 执行 `node scripts/test/prototype-smoke.js`。现有测试全部通过，证明脚本可初始化、主要画布可渲染、已覆盖页面无运行时异常及项目数据没有已知串台。

### 限制

- 本轮 in-app Browser 未提供可用浏览器实例，无法完成真实点击、截图、焦点流和移动端视觉验证。响应式结论来自 CSS/DOM 静态证据。
- 现有 smoke 使用模拟 DOM，能发现初始化、渲染和部分状态错误，但不验证真实布局、键盘操作、文本溢出，也没有覆盖 Agent 关键词路由、行级数据权限、报价/批量发信阈值。
- 本报告审查的是产品原型，不评价后端可行性、真实第三方数据质量、支付 partner 合同或生产级安全实现。

### 严重级别口径

- `P0`：阻断原型冻结；会使产品责任、权限、核心工作流或定位本身失真。
- `P1`：进入开发前必须补齐；否则关键用户任务无法被完整验证。
- `P2`：不阻断方向判断，但会降低可用性、可信度或开发规格完整性。

## 4. 结论计分板

| 维度 | 结论 | 评价 |
|---|---|---|
| AI-native 主交互 | 强 | 常驻 Agent + 画布已成为主入口，传统 UI 退居辅助 |
| 决策智能与可解释性 | 强 | 组合决策、规则审计、来源与可信度解释是当前最有差异化的部分 |
| 六环节业务闭环 | 部分 | 页面完整，但多数阶段门没有状态前置条件 |
| 多角色协作 | 弱 | 角色叙事明确，实际数据边界和动作归属不一致 |
| 人类闸门与审批 | 弱 | 交接 UI 完整，但审批未绑定业务生效，两个阈值未实现 |
| 全球化产品能力 | 弱 | 有多市场样例，缺少全球运营所需的本地化工作流 |
| 游戏垂类差异化 | 强 | Key、Mission、交付、partner 结算形成明确垂类深度 |
| 异常与恢复体验 | 弱 | 有空态和解析中，缺失败、重试、部分成功和外部动作失败 |
| 移动端可用性 | 弱 | 小屏隐藏导航，同时常驻 dock 覆盖大部分画布 |
| 作为开发蓝本的完整度 | 不足 | 多个看似可操作的入口实际上不改变业务状态 |

### 已经达到重构目的的部分

1. **主轴确实换了。** 右侧 Agent 常驻，意图经统一入口分发，结果在中央画布呈现，而非散落的“AI 按钮”（原型 769-792、2096-2211 行）。
2. **人的工作从逐条操作上升为组合决策。** Match 用“选 1 个组合 + 判断少量例外 + 抽样审计”控制规模增长，而不是继续翻万行表（1789-1921 行）。
3. **可信度设计有实质差异化。** 受众、可信评分、异常信号和逐字段来源被放在同一判断面板中（930-958、1290-1318、1939-1953 行）。
4. **游戏垂类不是口号。** Key 分发、Mission、内容审核和 partner 结算已进入六环节，而不是孤立功能页（1464-1530、1617-1648 行）。
5. **人类闸门理念正确。** 原型区分不可逆外部动作和可撤销内部动作，这比“所有动作都弹确认框”成熟（2234-2279 行）。
6. **责任链被显性化。** 交接单能显示发起人、待办人、批准、退回和重提，而不是只有状态标签（1251-1279、1356-1375 行）。

## 6. 结构化问题清单

### GAP-001 产品到底是“全球 KOL 平台”还是“全球游戏 KOL 平台”尚未锁定

- 严重级别：`P0`
- 页面/模块：产品定位 / 全局
- 原型期望：在冻结原型前锁定 ICP、行业边界和核心不可替代价值，避免第二次路线偏移。
- 当前实际：用户表述是“全球 KOL 营销平台”；仓库结论已把差异化收窄为“游戏垂类 Agent 工作流”，原型中的 campaign、Key、Steam、Mission、游戏合规和全部样例也都针对游戏行业。
- 差距类型：产品策略
- 影响：如果目标是横向 KOL 平台，当前抽象被游戏场景过度定义；如果目标是游戏垂类，当前对外定位又不够明确。两种选择会直接改变模块优先级、数据源、销售对象和全球化要求。
- 证据：`gap-product-layer.md` 98-110 行；原型 827-911、1138-1157 行。
- 相关文件：`docs/product/gap-product-layer.md`、`docs/product/interaction-prototype.html`
- 建议移交：产品负责人必须在下一轮原型前明确二选一，并写成一句不可歧义的定位与首批 ICP。建议倾向当前证据更强的“AI-native 全球游戏创作者营销平台”。

### GAP-002 六环节的“阶段门”除 Brief 外没有业务守卫

- 严重级别：`P0`
- 页面/模块：项目 / 六环节
- 原型期望：每个阶段门都是责任交接点，只有前置成果完成后才能推进；越门需明确风险和留痕。
- 当前实际：`gateAdvance` 只检查 Brief 覆盖度。发现、匹配、沟通、交付都可直接 `c.stage++`。当前沟通环节仍有待处理条款也可推进；交付检查中的 `#ad` 未通过也不阻断；候选池未生成组合时也能进入智能匹配。
- 差距类型：状态 / 工作流
- 影响：六环节会退化为可点击进度条，责任链和合规门失去真实性；后续实现很容易据此做出错误状态机。
- 证据：原型 1550-1565、1605-1633、1664-1672、2274-2279 行。
- 相关文件：`docs/product/interaction-prototype.html`
- 建议移交：为 01-06 分别定义 `entryCriteria`、`exitCriteria`、阻断项、可越门角色和越门理由；至少演示沟通未完成、披露未通过、组合未批准三种阻断。

### GAP-003 结算动作与生命周期顺序冲突，可在交付前放款

- 严重级别：`P0`
- 页面/模块：沟通协作 / 内容交付 / 结算
- 原型期望：放款条件“内容审核通过”必须由交付环节的真实状态驱动，且不可逆动作只在条件满足后出现。
- 当前实际：`gsea` 当前停在 04 沟通协作，05 内容交付按状态仍“尚未进入”，但 04 的结算面板已显示 Ayu 内容审核通过并允许财务放款。
- 差距类型：状态 / 数据映射 / 资金风险
- 影响：不可逆资金动作建立在互相矛盾的阶段状态上，破坏用户对 Agent 和审批链的信任。
- 证据：原型 855-859、1152-1157、1510-1530、1554-1560 行。
- 相关文件：`docs/product/interaction-prototype.html`
- 建议移交：重新决定“结算归属 04”是组织归属还是生命周期位置。可保留 finance owner，但付款条件应消费 05 的审核事件，并在项目时间线上显示跨阶段依赖。

### GAP-004 审批链没有绑定业务生效，且两个量化阈值未执行

- 严重级别：`P0`
- 页面/模块：Match / Reach / 驾驶舱审批
- 原型期望：组合 > `$8,000`、报价 > `$2,000`、批量发信 > `10` 自动转审批；批准前业务对象不可用于下游；退回原因由决策人填写。
- 当前实际：只有组合预算阈值被引用；`THRESHOLD.quote` 与 `THRESHOLD.bulkSend` 从未使用。BD 选 `$9,800` 方案后，`selectedPlan` 已立即生效，Reach 可据此起草和发送；批准动作只改审批单状态。`$2,400` 报价可通过条款弹窗直接发出。退回原因固定为一段硬编码文本。
- 差距类型：权限 / 状态 / 审批
- 影响：审批是视觉闭环而非业务闭环，用户可以在待批状态完成本应被阻断的外部动作。
- 证据：原型 992-1000、1063-1073、1251-1279、1963-1967、2229-2233、2299-2308、2339-2345 行。
- 相关文件：`docs/product/interaction-prototype.html`
- 建议移交：审批单必须引用目标业务对象与待执行 action；批准后原子生效，退回保持不可用。补齐 quote/bulkSend 两条可操作样例和可编辑退回原因。

### GAP-005 角色边界在数据级和动作级失效，默认主路径还会误拦截

- 严重级别：`P0`
- 页面/模块：角色切换 / 驾驶舱 / 项目 / Reach / Delivery
- 原型期望：BD 只看负责项目且不看全局预算；Lead 不执行 BD 的发信和 Key 分发；所有入口按同一权限规则工作。
- 当前实际：BD 的项目页展示全部项目、总预算和跨项目 ROI；项目作用域可切日韩/全球项目。Lead 访问 Reach/Delivery 时仍能看到发信、条款确认和 Key 分发按钮，因为这些动作没有 `ownsGate` 守卫。默认角色是 BD，而首屏“建项目”提示词包含“预算”，会先被 BD 的预算关键词拦截，无法进入创建草稿分支。
- 差距类型：权限 / 主流程
- 影响：角色体系是本次重构的核心地基，但当前原型会同时造成越权暴露、越权动作和合法任务误拒绝。
- 证据：原型 965-989、1182、1428-1457、1488、1966-1987、2067-2082、2101-2112、2179-2183、2366-2374 行。
- 相关文件：`docs/product/interaction-prototype.html`
- 建议移交：权限模型拆成模块权限、对象范围、字段遮罩、动作权限四层；所有 UI 与 Agent 工具消费同一 policy。新增三角色正向/反向任务矩阵，而非只测“能否打开模块”。

### GAP-006 “AI 起草、人审阅、可修正”多处仍是假动作

- 严重级别：`P1`
- 页面/模块：Brief / 知识库 / Reach / 关系资产
- 原型期望：用户可用自然语言或卡片精调 Agent 草稿，修改后画布和下游结果同步变化。
- 当前实际：Brief 的“改”、知识卡“修正”、邮件“编辑”、添加团队判断等入口只给提示或 `data-noop`；后续输入“预算改成 1.5 万”“受众改成……”没有对应状态修改分支。对话精调只回复“已重排”，组合数据并未变化。
- 差距类型：交互 / 状态
- 影响：用户只能接受固定假数据，无法验证 AI-native 产品中最关键的人机共创循环。
- 证据：原型 1421-1425、1732-1737、1916、1980-1981、2088、2185-2189、2348-2354、2375 行。
- 相关文件：`docs/product/interaction-prototype.html`
- 建议移交：选择 3 个高价值修改做真即可：Brief 预算/市场、知识卡人工修正、邮件正文。每次修改必须展示 diff、来源、影响范围和可撤销状态。

### GAP-007 Match 到 Reach 的对象数量和状态不一致

- 严重级别：`P1`
- 页面/模块：Match → Reach
- 原型期望：选定组合后，下游严格消费同一批 KOL，数量、成员、审批状态和邮件草稿一一对应。
- 当前实际：Agent 可宣称已为 14 位组合成员起草邮件，但 Reach 始终只渲染固定的 3 封 `EMAILS`。发送状态又是一个全局 `emailsSent` 布尔值，单封发送会让全部草稿变成已发送。
- 差距类型：数据映射 / 跨模块状态
- 影响：用户无法验证最重要的跨模块承诺，开发团队也无法从原型确定 campaign-roster-draft-send 的实体关系。
- 证据：原型 872-881、1063-1073、1180、1976-1987、2192-2203、2225-2228 行。
- 相关文件：`docs/product/interaction-prototype.html`
- 建议移交：以 campaign roster 为唯一来源生成 draft 列表；单封、批量、失败和重发都应有逐项状态，不使用全局布尔值。

### GAP-008 “信号驱动 CRM”没有被原型证明

- 严重级别：`P1`
- 页面/模块：Reach / CRM
- 原型期望：送达、打开、回复、退信、签约等真实信号自动推进 CRM，并触发 Agent 下一步建议与抑制名单。
- 当前实际：回复数据是预置数组，发送后只把 `emailsSent` 改为 true；没有送达/打开/退信/抑制状态，没有从回复或签约事件推导 CRM 状态的状态机。
- 差距类型：状态 / 工作流
- 影响：原型仍然只展示“AI 写邮件”，没有证明从传统手工 CRM 到信号驱动运营的关键跃迁。
- 证据：`ai-native-usage.md` 153-175 行；原型 883-895、1957-1988、2225-2233 行。
- 相关文件：`docs/product/ai-native-usage.md`、`docs/product/interaction-prototype.html`
- 建议移交：增加一条可演示事件链：发送 → 部分送达/1 封退信 → 1 位回复 → CRM 自动推进 → Agent 起草回信 → 高报价转审批。

### GAP-009 Insight 的周报和对外分享没有生成式画布与闸门

- 严重级别：`P1`
- 页面/模块：Insight
- 原型期望：Agent 生成完整周报卡片，用户审阅后导出 PDF 或经确认生成/撤销分享链接。
- 当前实际：点击生成周报后只在对话气泡里返回一句摘要，画布不产生周报，PDF、分享、确认和撤销都没有交互。
- 差距类型：交互 / 人类闸门
- 影响：Insight 没有完成“Agent 交付工作成果”的闭环，仍接近静态看板 + 问答。
- 证据：`ai-native-usage.md` 179-199 行；原型 2002-2038、2158-2161 行。
- 相关文件：`docs/product/ai-native-usage.md`、`docs/product/interaction-prototype.html`
- 建议移交：补一个完整周报 artifact，至少包括来源范围、可编辑摘要、导出、分享确认、链接权限与撤销。

### GAP-010 “全球”目前主要由样例市场表达，尚未形成全球运营工作流

- 严重级别：`P1`
- 页面/模块：Brief / Reach / Insight / 合规
- 原型期望：全球团队能处理语言、本地审阅、时区、币种、市场披露规则和跨区域责任范围。
- 当前实际：样例覆盖东南亚、日韩、北美和全球，但没有本地币种/汇率、时区排期、翻译审阅、本地审批或逐市场合规流程。泰国邀约邮件出现泰语问候 + 中文正文，说明本地化仍是展示文本而非工作流。
- 差距类型：产品能力 / 全球化
- 影响：“全球”容易退化为地区字段和多语言假数据，无法验证真实跨市场团队如何协作。
- 证据：原型 850-895、1086-1102、1172-1174 行；全文件未定义 timezone/currency/locale 工作流。
- 相关文件：`docs/product/interaction-prototype.html`
- 建议移交：选一个跨市场 campaign 做黄金路径，至少包含基准币种 + 本地币种、时区排期、双语草稿/本地 reviewer、市场披露检查和区域 scope。

### GAP-011 Assets 模块在产品使用方式中存在，但原型完全缺席

- 严重级别：`P1`
- 页面/模块：Assets / Reach

- 影响：已确认的产品范围没有被原型覆盖，后续开发无法判断 Assets 是独立模块、上下文 artifact，还是应被取消。
- 证据：`ai-native-usage.md` 203-213 行；原型 705-740、2067-2082 行。
- 相关文件：`docs/product/ai-native-usage.md`、`docs/product/interaction-prototype.html`

### GAP-012 小屏布局没有可用的导航和 Agent 开合机制

- 严重级别：`P1`
- 页面/模块：全局响应式
- 原型期望：移动/窄屏至少能在画布、导航和 Agent 之间切换，不发生永久遮挡。
- 当前实际：`≤860px` 时左侧 rail 直接隐藏，右侧 dock 固定占 `min(360px, 88vw)`，源码没有菜单按钮、dock 关闭按钮或显隐状态。
- 差距类型：响应式
- 影响：小屏上大部分画布会被 Agent 覆盖，同时无法切换模块。真实视觉尚未运行验证，但源码行为已构成明确高风险。
- 证据：原型 680-688、698-792 行。
- 相关文件：`docs/product/interaction-prototype.html`
- 建议移交：将移动端定义为三态工作区：导航抽屉 / 画布 / Agent sheet，保留明确开关和返回路径；完成 390px、768px、1440px 实机截图验收。

### GAP-013 Agent 与外部动作缺少失败、重试和部分成功状态

- 严重级别：`P1`
- 页面/模块：全局 / Agent / Reach / 支付 / Key
- 原型期望：AI 原生产品必须让用户看见工具调用失败、超时、数据不足、部分发送、权限拒绝和安全恢复路径。
- 当前实际：有空态、权限拒绝和知识库“解析中”，但没有解析失败、搜索无结果后的改写建议、邮件部分失败/退信、partner 放款失败、Key 分发部分成功或重试。
- 差距类型：状态 / 信任
- 影响：原型只验证 happy path，无法判断 Agent 出错时是否仍比传统 SaaS 更可控；这会直接影响用户信任设计。
- 证据：原型 1697-1715、1957-1988、2217-2273 行；错误关键词扫描无对应状态。
- 相关文件：`docs/product/interaction-prototype.html`
- 建议移交：每类外部动作至少设计“进行中 / 部分成功 / 失败可重试 / 需人工接管”四态，并显示已完成与未完成对象。

### GAP-014 传统 UI 作为“精调兜底”的承诺没有落地

- 严重级别：`P2`

- 原型期望：自然语言是主入口，但专业用户可以打开精调抽屉设置精确地区、平台、粉丝区间等条件。
- 当前实际：只有对话精调和排除规则审计，没有可展开的精调抽屉。
- 差距类型：交互 / 兜底能力
- 影响：复杂采购条件无法精确表达，也无法验证 Agent 解析结果与人工约束的冲突处理。
- 证据：`ai-native-usage.md` 128-147 行；原型 1888-1921 行。

### GAP-015 数据有来源但没有新鲜度，可信链条还差最后一环

- 严重级别：`P2`
- 页面/模块：KOL 经营档案 / Match

- 差距类型：数据映射 / 信任
- 影响：全球 KOL 数据变化快，用户无法判断评分和受众画像是否仍可用于花钱决策。
- 证据：`gap-data-layer.md` 80-100 行；原型 930-958、1939-1948 行。

### GAP-016 可访问性和次要控件尚未形成完整原型契约

- 严重级别：`P2`

## 11. 结论与建议路线

### 是否达到了重构目的

**方向上达到了，工程蓝本上尚未达到。**

当前原型已经找到了一个比旧 kolmatrix 更值得开发的产品形态：不是让用户更快地操作数据库，而是让 Agent 承担搜索、组合、起草、归纳和分析，让人负责规则、例外、预算和对外承诺。尤其是“组合级决策 + 可解释评估 + 游戏垂类六环节 + 关系资产”已经形成一条有竞争辨识度的主线。

但原型目前仍大量依赖文案宣称状态变化，而不是让状态变化真的发生。对于这次“先把产品和工作流磨清楚，再开发”的策略而言，这些不是可以留给开发补齐的细节，恰恰是原型阶段必须先解决的产品问题。

### 建议的原型冻结门槛

1. 锁定“游戏垂类 / 横向平台”定位和首批 ICP。
2. 让六阶段每个阶段至少有一个真实退出条件和一个阻断案例。
3. 让组合、报价、批量发信三条审批与业务对象真正绑定，批准前无法进入下游。
4. 用同一套 policy 同时驱动 Agent、画布、对象范围、字段和动作权限。
5. 做通一条完整黄金路径：Brief 可修改 → Match 组合审批 → 对应人数邮件 → 回复信号 → 报价审批 → 交付检查 → finance 放款 → Insight 周报。
6. 做通一个全球化样例和一个失败恢复样例。
7. 完成桌面和移动端真实浏览器验收，并扩充 smoke 的业务语义断言。

### 最终建议

- 是否建议进入修复：**是，继续原型修订。**
- 是否建议现在进入全面开发：**否。** 可以继续进行不依赖产品细节的 Agent/权限基础设施，但不应按当前页面和状态模型批量实现业务模块。
- 是否建议重新验收：**是。** P0 全清、P1 至少完成核心黄金路径后重新审查。
- 备注：本次仅输出审查结果，未做任何产品代码或原型修改。
