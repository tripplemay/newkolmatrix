# KOLMatrix 产品功能模块使用方式（AI-native 重构版）

> **用途：** 给用户审阅，确认 Planner 对需求的理解是否正确。**这是产品使用方式文档，不是技术架构文档。**
> **状态：** 待用户确认（确认后据此校准 AGENT-FOUNDATION 及后续批次的 spec）
> **日期：** 2026-07-14

---

## 0. 总纲：这个产品"怎么用"变了

**旧系统（传统 SaaS）的用法：** 用户像操作数据库工具一样用产品——自己点菜单找模块、手填 15 维筛选、翻表格淘 KOL、手动填活动表单、手动切 CRM 状态、手动拼周报。AI 是散落在各页角落的"加分按钮"。

**新系统（AI-native / Agent 驱动）的用法：** 产品中心是一个**常驻的 Agent 对话面**。用户**表达意图**（自然语言说要什么），Agent **主动干活**（调用工具：搜 KOL、建活动、起草邮件、算 ROI、生成报告），结果以**卡片/面板的形式"长"在画布上**（generative UI）。用户的角色从"操作员"变成"决策者 + 审阅者"。

### 三类角色分工（贯穿所有模块）
| 谁 | 干什么 |
|---|---|
| **用户** | 表达意图、审阅 Agent 产出、对高风险动作点确认、必要时手动精调 |
| **Agent** | 理解意图 → 规划 → 调工具 → 把结果渲染到画布 → 解释"为什么" |
| **传统界面（表单/表格/筛选器）** | 退为**兜底/精调**——想手动细调时才展开，不再是入口 |

### 五条使用方式原则（每个模块都遵守）
1. **一句话代替一堆勾选**：需求用自然语言说，Agent 解析成结构化操作。
2. **推荐流代替翻页表格**：Agent 主动给"排好序 + 带理由"的 Top-N，用户 Accept/Skip，而非在万行表里手动淘。
3. **AI 起草、人审阅**：活动、邮件、报告由 Agent 起草整份，用户改而非从零填。
4. **信号驱动代替手动打标**：CRM 状态从真实事件（已回复/已签约）自动推断，Agent 建议下一步。
5. **高风险动作留人类闸门**：发邮件、录营收/花钱这类不可逆或对外动作，Agent 准备好，**必须用户点确认才执行**。

### 一个跨模块的用法示例（体现 Agent 能串起多个模块）
> 用户在对话面打字：**"给我上周为『原神东南亚』活动推荐的、粉丝 10 万以上的 KOL，起草一封触达邮件让我看看。"**
> Agent：① 调 `search`/`match` 取该活动上周推荐里粉丝≥10万的 → ② 卡片流呈现 → ③ 调 `draft_email` 用产品资料+KOL 画像生成个性化邮件草稿 → ④ 草稿卡片呈现，附「发送」按钮（**人类闸门**）。用户改两句 → 点发送 → Agent 调 `send_email` 发出并落追踪。

这一条就跨了 Match + Reach 两个模块，全程无需用户切页面、无需手填收件人/模板/变量。**这就是"去 SaaS 化"的样子。**

## 1. Brief — 意图输入（活动创建 + 产品知识库）

**职责：** AI-native 的入口。用户用自然语言描述"要推什么产品、给谁、预算多少"，Agent 把它变成一个结构化的营销活动 + 补全产品知识。

**AI-native 使用方式：** 不是打开一张空白活动表单逐字段填，而是对 Agent 说一句话，Agent 解析成活动草稿（名称/关联产品/预算/周期/目标市场/受众/类目）呈现在画布上，用户核对/微调后确认建立。产品知识库（我推什么游戏、卖点是什么）也由对话录入，作为后续 AI 匹配和文案生成的"燃料"。

**典型流程（用户视角）：**
1. 对话面输入："Q2 要给二次元手游《XX》推东南亚市场，预算 1 万美金，想找女性向、中腰部的游戏区创作者。"
2. Agent 解析 → 画布呈现活动草稿卡（已填好各字段）+ 若产品库没有《XX》则提示补一条产品信息。
3. 用户核对，改动某字段（可直接说"预算改成 1.5 万"或在卡片上编辑）。
4. 确认 → 活动建立 → Agent 直接引导进入下一步（"要现在就看 KOL 推荐吗？"）。

**旧 SaaS → AI-native 转变：**
| 旧（副驾） | 新（主驾） |
|---|---|
| 主路径 = 手填 CampaignForm 表单；自然语言解析器藏在角落且自称 "escape hatch" | 主路径 = 自然语言意图；表单退为"想手动改字段时"的精调面 |
| 产品库 = 独立的 CRUD 面板，手动增删改 | 产品信息在对话中随需补录，被 Agent 主动索取 |

**复用/放大的 AI 能力：** 旧系统的 `brief-actions`（自然语言→活动字段解析，已工程化落地）——从"逃生舱"提为主入口。

## 2. Campaigns — 活动管理

**职责：** 管理每个 KOL 营销活动的全生命周期（从建立 → 推荐 KOL → 接受合作 → 追踪 → 结算 ROI）。

**AI-native 使用方式：** 活动详情页是一块**画布**：顶部是活动摘要，主体是 **Agent 给这个活动的 KOL 推荐面板**（排好序、带匹配理由、可 Accept/Skip），下方是已接受的 KOL 及其合作状态。用户主要在这里做"决策"（接受/跳过推荐、录入费用/营收），而不是填一堆管理字段。Agent 还会给"下一步建议"（如"这 3 个 KOL 3 天没回复，要不要跟进"）。

**典型流程（用户视角）：**
1. 进入某活动 → 画布已经是 Agent 推荐的 Top-N KOL 流（无需手动搜）。
2. 逐个 Accept（加入活动）或 Skip（并可说原因，Agent 据此调整后续推荐）。
3. 合作推进中：录入谈定的费用；活动结束录入带来的营收 → 系统自动算 ROI。
4. 随时可问 Agent："这个活动现在进展怎么样？" → Agent 汇总状态 + 建议下一步。

**旧 SaaS → AI-native 转变：**
| 旧 | 新 |
|---|---|
| 活动列表 + 状态筛选器 + 手填编辑表单为主 | 活动详情=推荐决策画布；管理字段退居其次 |
| 合作状态（6 态）靠用户逐个手动切下拉 | 尽量从信号自动推断，Agent 建议状态推进 |

**复用/放大的 AI 能力：** `AiRecommendationPanel`（smart-match top-30）、`campaigns/suggestions`（next-action 建议）——从"卡片"提为活动画布主体。

**人类闸门：** 录入营收/费用（涉及金钱数据）需用户明确操作；Accept KOL = 用户决策。

## 3. Match — KOL 发现 + 评估工作台（产品心脏）

**职责：** 合并旧系统的"发现（探索新 KOL）"和"数据库（已存 KOL 池）"——一个让用户找到、评估、选择合适 KOL 的工作台。这是 AI-native 翻转最戏剧化、也是最能体现"去 SaaS 化"的地方。

**AI-native 使用方式：** 用户**用一句话描述想找什么样的 KOL**，Agent 做语义搜索 + 智能匹配，返回**候选 KOL 卡片流**——每张卡带匹配分和"为什么推荐它"的解释。用户 Accept/Skip，或继续用自然语言精调（"少点微网红、多点女性受众"），Agent 重排。那个 687 行、15 个维度的手填筛选侧栏**降级成一个可选的"精调抽屉"**——想手动卡粉丝区间/地区时才拉出来，不再是入口。KOL 画像详情（粉丝画像、近期视频、联系邮箱、合作历史）也是一块画布，用户可让 Agent"看看这个人适不适合我的活动"。

**典型流程（用户视角）：**
1. 对话面："找东南亚手游区、粉丝 10-50 万、最近活跃的女性向创作者。"
2. Agent → 语义搜索 + 匹配 → 画布呈现候选 KOL 卡片流（带分数 + 一句理由）。
3. 用户翻看，Accept 几个（可直接加入某活动），Skip 掉不合适的。
4. 觉得方向偏了 → 说"排除已经合作过大厂的、多给点素人感的" → Agent 重排。
5. 点某个 KOL → 画像画布：粉丝地域/年龄/性别分布、近期视频、主题、联系邮箱、valueScore、历史合作/沟通记录。
6. （可选）真要手动精卡某维度 → 拉出精调抽屉勾选。

**旧 SaaS → AI-native 转变：**
| 旧 | 新 |
|---|---|
| 入口 = 15 维手填筛选侧栏（用户得先懂"monetization status""channel age tier"字段本体） | 入口 = 一句自然语言；筛选侧栏退为精调抽屉 |
| 主界面 = 数据表格 + 游标翻页，用户在 10⁵ 行里手动淘 | 主界面 = Agent 推荐的 Top-N 卡片流 + 理由；全表退为"想看全部时才展开"的兜底 |
| "为什么这个 KOL 值得推"要用户自己判断 | Agent 主动给可解释性评分（匹配度/类目契合/近期活跃/受众契合/品牌历史） |

**复用/放大的 AI 能力（这块旧系统资产最厚）：** bge-m3 向量嵌入、语义搜索、SmartMatch（product→cosine top-30）、LLM 重排 + 15 词匹配理由、可解释性评分（1 句 short + 5 段 detailed）——全部已工程化落地并带优雅降级，本重构把它们从"辅助"提为交互主轴。**（AGENT-FOUNDATION 的 hello-agent `search_kols` 就是这块的最小起点。）**

## 4. Reach — 触达 + CRM

**职责：** 选定 KOL 后，把"联系他们"这件事从碎片化的手动 Gmail 发信，变成 AI 起草 + 一处发送 + 自动追踪 + 关系管理。

**AI-native 使用方式：** 用户不再打开一个 1260 行的邮件工作台手动配收件人/选模板/填变量。而是对 Agent 说"给这批 KOL 起草触达邮件"，Agent 结合产品资料 + 每个 KOL 的画像，**为每个人生成一封个性化邮件草稿**，呈现在画布上供审阅。用户改完点发送（**人类闸门**），系统经 Resend 发出并自动追踪状态（送达/打开/回复/退信）。CRM 不再靠用户逐个手动切"已联系/已谈判/已签约"下拉——状态尽量**从真实信号自动推断**（收到回复→自动标记），Agent 主动提示"这几个该跟进了"。退订/退信自动进抑制名单（合规）。

**典型流程（用户视角）：**
1. 从 Match/活动里选好一批 KOL → 对话面："给这批人起草触达邮件。"
2. Agent → 逐个生成个性化草稿（主题+正文+为什么这么写的说明）→ 画布卡片流呈现。
3. 用户逐封审阅、微调（或说"整体再热情一点/加上限时福利"，Agent 批量改）。
4. 点「发送」（**闸门**）→ 经 Resend 发出 → 画布切到发送追踪视图。
5. 有人回复 → CRM 状态自动推进 → Agent 提示"KOL A 回复了，要不要起草回信？"

**旧 SaaS → AI-native 转变：**
| 旧 | 新 |
|---|---|
| 1260 行手动邮件工作台：选模板→填变量→发送 | Agent 起草整封个性化邮件，人只审阅微调 |
| CRM 靠用户逐个 KOL 手动拖状态 | 状态从信号（已回复/已签约事件）自动推断 + Agent 建议下一步 |
| 模板库/追踪/抑制名单各一个独立子页，用户"去哪个页做哪件事" | 功能随触达对话上下文出现（在对话里直接看追踪、调模板） |

**复用/放大的 AI 能力：** `email/customize`（模板×产品×KOL→个性化改写）、`campaigns/suggestions`（下一步建议）、对话式 Refine——从"AI 定制按钮"提为触达主流程。

**人类闸门：** **发送邮件必须用户确认**（对外、不可逆）。批量发送尤其要显式确认数量。

## 5. Insight — 数据看板 + ROI + AI 周报

**职责：** 复盘（我的营销做得怎么样）+ 对外汇报（给老板/客户看的周报）。

**AI-native 使用方式：** 看板不只是静态图表堆砌——用户可以**问**："这个月哪个活动 ROI 最高？""触达率为什么降了？"Agent 读数据 + 给出带洞察的回答，并把相关图表渲染到画布。ROI 自动计算（营收−KOL 费用）/费用，Agent 附"AI 洞察"文案。周报**一键让 Agent 生成**（读 CRM/ROI/进展 → 生成 markdown 周报 → 可导 PDF / 生成分享链接给客户看），免去手工拼 PPT。

**典型流程（用户视角）：**
1. 进 Insight → 画布默认呈现 KPI + 工作流进度 + 活动流 + ROI 趋势。
2. 用户问 Agent："上周表现总结一下，重点说下哪些活动值得加投。" → Agent 汇总 + 建议。
3. 要给客户汇报 → "生成本周周报" → Agent 生成周报 → 用户审阅 → 导 PDF 或生成公开分享链接。
4. ROI 页：录入营收后自动算，Agent 给洞察（"活动 A 的 ROI 3.4x，主要来自 3 个腰部 KOL"）。

**旧 SaaS → AI-native 转变：**
| 旧 | 新 |
|---|---|
| 看板 = 固定图表；用户自己看数字找结论 | 看板可对话追问，Agent 给洞察 + 按需渲染图表 |
| 周报虽 AI 生成但是独立功能页 | 周报是 Agent 在对话里随手交付的产物 |

**复用/放大的 AI 能力：** AI 周报生成（gemini 长上下文）、ROI insights、KPI 快照——继续用，且入口对话化。

**人类闸门：** 生成公开分享链接（对外暴露数据）= 用户确认；撤销分享链接随时可做。

## 6. Assets — AI 素材库

**职责：** 从产品资料生成可复用的触达素材（邮件模板、短视频脚本）。

**AI-native 使用方式：** 用户说"给《XX》生成 3 版触达邮件和 2 个短视频脚本"，Agent 生成 → 素材卡片呈现 → 用户挑选/编辑 → 可直接送到 Reach 用于触达。素材有变体树（一版衍生多版）。

**典型流程：** 对话请求 → Agent 生成素材 → 画布卡片 → 编辑/采纳 → 一键用于触达。

**旧 SaaS → AI-native 转变：** 从"素材列表 + 生成向导 + 预览抽屉"的管理界面，变为对话请求 + 画布呈现 + 直接接入下游触达。

**复用/放大的 AI 能力：** `assets/generators`（邮件/视频脚本生成器）。

## 7. 数据来源 — KOL 库怎么进来（后台，用户基本无感）

**职责：** KOL 数据的摄入与维护（这是"库里为什么有 KOL 可搜"的来源）。

**使用方式：** 用户**不**手动 CSV 导入导出、不手动运维数据。数据由后台自动摄入：
- **本重构初期（AGENT-FOUNDATION 及前几期）：** 用旧系统已积累的 ~2500 条真实 KOL（CSV seed）灌库 + 生成语义向量，让 Agent 立刻有真实数据可搜。
- **后期（Phase 5）：** 接回 Apify 采集管道（YouTube 等平台每日增量爬取 + AI 富化国别/类目），持续扩库。
- Admin 运维页（导入监控、爬虫监控）是给管理员的后台工具，普通营销用户看不到。

**旧 SaaS → AI-native 转变：** 从"用户手动 CSV 批处理搬数据"，变为"数据自动流入，用户只面对结果（能搜到的 KOL）"。

## 8. 认证 / 邀请制 / 落地页（外围，简述）

- **落地页**：匿名访客获客转化（Hero/痛点/功能/试用引导），留资走埋点。**本重构初期不是重点**，可后期批次还原。
- **邀请制访问**：无自助注册，填申请表 → 管理员审批建账号 → 登录。**本重构初期单租户、暂缓真实认证**（AGENT-FOUNDATION 用硬编码 dev 租户），认证/多租户留后期批次。

## 9. 请你重点确认的几处（我做了判断的地方）

以下是我基于"保留功能 + 去 SaaS 化 + 激进 Agent 驱动"做的**理解判断**，最需要你确认对不对：

1. **交互主轴 = 常驻 Agent 对话面 + generative canvas**（工作流页面是 Agent 产出的"落地画布"，不是独立操作台）。这是"激进 Agent 驱动"的核心理解——**对吗？** 还是你想要"页面为主 + Agent 为辅的对话侧栏"这种更温和的形态？
2. **筛选器/表格/表单一律降级为"精调兜底"**，不再是入口——**认可吗？** 有没有哪个模块你希望保留传统界面为主（比如财务/ROI 这种精确录入场景）？
3. **人类闸门设在：发邮件、录营收/费用、生成对外分享链接**。**够吗？** 还有哪些动作你要求必须人工确认？
4. **模块范围与优先级**：Brief→Campaigns→Match→Reach→Insight 五大 + Assets + 数据摄入 + 认证/落地页。**有没有旧系统的功能我漏了、或有你想砍掉不重构的？**
5. **CRM 状态自动推断**：我理解成"尽量从信号自动推进 + Agent 建议"，但保留手动改的兜底。**这个程度对吗？**
6. **初期单租户 + CSV seed 真实数据 + Apify 留后期**——这个务实取舍你接受吗？

请审阅，指出任何理解偏差。你确认后，我会据此校准 spec（若有偏差需要改 AGENT-FOUNDATION 的范围，我会同步调整 features.json）再进入 building。
