# 差距分析 · 数据层

> **本层定义：** KOL 数据**从哪来、有多少、有多深、多合规、多少钱**。
> **本层特征：** **外部依赖重**——要花钱（年付合约）、要谈准入（平台伙伴）、要供给侧运营（创作者入驻）。**不由我们的代码决定**，推进节奏受外部约束。
> **本层状态：`暂缓`**（用户 2026-07-14 决策：现阶段先聚焦产品功能层）。本文档负责**把方向和接口先定下来**，使功能层不被本层阻塞。
> **姊妹文档：** 证据 → [`competitive-research-evidence-2026-07.md`](./competitive-research-evidence-2026-07.md) · 功能层 → [`gap-product-layer.md`](./gap-product-layer.md)

## 2. 四条获取路径对比

| 路径 | 优点 | 缺点 | 成本锚点 |
|---|---|---|---|
| **爬取**（现状） | 零成本起步、可控 | 字段浅（无受众/可信度）；**合规风险**（TikTok 已封关键词追踪）；中期不可持续 | 基建人力 |
| **外购 API** | **最快**，一步买齐受众画像 + 可信度；亿级覆盖 | 年付承诺无月付；**Modash 无 Twitch**；游戏垂类质量未验证；数据非独占（对手同源） | **Modash：$10k–16.2k/年起**<br>（Discovery $16,200 / Raw $10,000） |
| **创作者 opt-in 入驻** | 数据真实、合规、**可要授权**；可要邮箱/报价；**垂类对手都这么做**（护城河） | **慢**；供给侧鸡生蛋；需运营投入 | 供给侧运营人力 |
| **平台一方 API** | 最权威、最合规、规模最大 | YouTube **仅入选伙伴、无自助通道**（17 家均大厂）；TikTok 需走 TTCM；准入不确定 | 商务/准入成本 |

## 3. 合规风险时钟（爬取路线的中期威胁）

| 平台 | 动作 | 对我们的影响 |
|---|---|---|
| **TikTok** | 已禁第三方用关键词检测追踪 campaign 帖子 → 强制 TTCM API 发订单；旧采集帖将被隐藏（Brandwatch 2026-02 实证 + 生态佐证，**证据为 medium**） | ⚠️ **效果追踪环节风险最高** |
| **YouTube** | BrandConnect → Creator Partnerships，API 仅对 select partners 开放、无自助 | 一方数据有 partner 路径但准入难 |
| **Twitch** | 一方 Sponsorships 入口（Dashboard 内），但结算交由 StreamElements 等伙伴 | **执行/结算层仍有 partner 空间** |

**结论：** 平台方正在把 KOL 市场收归一方并封堵第三方爬取。**发现环节**的爬取还能撑一段，**效果追踪环节**（P4 Insight）最先撞墙——那里必须预留一方授权架构位。

## 4. 建议策略（方向已定，执行暂缓）

**混合模式**，四条腿并行、按阶段接入：

1. **短期**：维持爬取做发现底座（够 demo、够早期客户）
2. **中期**：**外购通用数据**补齐受众画像 + 可信度（P0 的 buy 路径，$10k-16.2k/年起）
   - ⚠️ **Twitch 缺口**：Modash 无 Twitch → 需 HypeAuditor（含 Twitch）或自建补源
3. **中长期**：**自建游戏垂类信号**（外购数据在游戏垂类的质量未验证，通用 credibility 未必适用游戏创作者）
4. **长期护城河**：**创作者 opt-in 入驻**（垂类对手的真正壁垒）+ **争取平台一方 API 准入**

## 5. ⭐ 与产品功能层的接口契约（**本文档最重要的一节**）

**这是让两层解耦的核心机制**：产品功能层只依赖**字段契约**，不依赖数据来源。数据层无论走哪条路径，产出都落到同一组字段上——**功能层现在就能用 mock / 部分真实数据把 UI 和交互建完，数据到位后填真值即可，无需返工。**

### 5.1 字段契约（Phase 0 schema 预留位，本批不填充）

```
Kol {
  // ── 现有：爬取即可得（已有）
  platform, handle, name, channelUrl, region,
  followerCount, categories[], isGaming, aiReasoning

  // ── 数据层契约位（Phase 0 预留 nullable，数据层到位后填充）
  audienceDemo: {          // 受众人口统计 —— table-stakes
    ageDist,               //   年龄分布
    genderDist,            //   性别分布
    geoDist,               //   地域分布
    interests[]            //   受众兴趣
  } | null

  credibility: {           // 可信度 / 假粉 —— table-stakes
    score,                 //   0-100（对齐 GRIN 口径）或 0-1（Modash 口径）
    signals[],             //   评分依据：互动率/粉丝比、增长模式、受众构成、点赞者可信度
    assessedAt
  } | null

  brandSafety: {           // 品牌安全（原型已有 UI，数据待接）
    rating,                //   A / A- / B+ …
    knownCollabs[],        //   历史品牌合作（竞品检测用）
    flags[]
  } | null

  engagement: {
    rate, avgViews, authenticity
  } | null

  // ── 溯源（关键，见 §5.2）
  dataSource: 'crawl' | 'purchased' | 'optin' | 'platform_api'
  dataFreshness: timestamp
  fieldProvenance: { [field]: source }   // 逐字段溯源
}
```

### 5.2 `dataSource` / `fieldProvenance` 为什么是关键设计

它让产品层能**诚实地**展示数据可信度，而不是把爬取推断和平台一方数据混作一谈：

- 爬取推断的受众画像 → 标「推断值，未验证」
- 外购 API 的 credibility → 标「第三方评估」
- 平台一方/创作者授权 → 标「平台一方数据 ✓」

**这本身是一个差异化机会**：竞品的评估分数多为黑盒（HypeAuditor 95.5% 是自报、有用户争议）。我们做**可解释 + 可溯源的评估**，与「可解释匹配」的产品主张一脉相承。详见功能层 P0-F1。

### 5.3 解耦效果

| | 数据层未到位时 | 数据层到位后 |
|---|---|---|
| 评估面板 UI | ✅ 可建（mock/部分真实 + 溯源标注） | 填真值，UI 不变 |
| 组合建议消费受众结构 | ✅ 可建（用爬取推断值） | 换数据源，逻辑不变 |
| 可信度评分卡 | ✅ 可建（标"待接入"或用推断） | 填真值 |
