## 2. 功能层优先级

### 🔴 P0-F1 · 评估面板（受众画像 + 可信度评分）

**为什么 P0：** 受众人口统计 + 假粉/可信度评分是**全行业 table-stakes**——GRIN 甚至把它做成**免费获客工具**（0-100 分，四平台，无需账号）。**客户第一眼就看这个**。我们的 KOL 卡目前只有品牌安全徽标，没有受众结构、没有可信度。

**关键洞察：这是功能缺口，不是数据缺口——两者要分开做。**
- **功能层（现在做）**：评估面板的 UI、交互、可解释性设计
- **数据层（暂缓）**：真值从哪来（外购 API）

按 [数据层 §5 字段契约](./gap-data-layer.md#5--与产品功能层的接口契约本文档最重要的一节) 建：先用推断值/mock + **溯源标注**，数据到位后填真值。

**建议形态：**
- KOL 画像页新增**受众结构**（年龄/性别/地域分布图表 + 受众兴趣）
- **可信度评分卡**（0-100）
- ⭐ **可解释评估**（差异化）：不只给分，还给**评分依据**（互动率/粉丝比、增长模式异常、受众构成、点赞者可信度）+ **数据溯源**（推断值 / 第三方评估 / 平台一方 ✓）
- 组合建议消费受众结构（如"越南缺口"可由受众地域分布驱动）

**差异化机会：** 竞品评估多为**黑盒**（HypeAuditor 95.5% 为自报、有 Trustpilot 争议）。我们做**可解释 + 可溯源**的评估，与"可解释匹配"一脉相承。

---

### 🟠 P1-F1 · game key 分发

**为什么 P1：** **游戏行业 table-stakes**——由 Keymailer / Lurkit / Woovit / Terminals **4+ 家平台构成的成熟独立品类**（Cloutboost 对比文佐证）。Lurkit：「distribute **thousands of game keys** to targeted creators」。我们六环节**完全未覆盖**。
**方式：`build`**（与交付环节天然衔接，非外部依赖）
**建议形态：** key 批次导入/生成 → 定向分发（按候选/组合）→ 领取与兑换追踪 → 与内容交付环节联动（发了 key 是否产出内容）→ Agent 可调用的工具

---

### 🟠 P1-F2 · 绩效计费机制（Quests / Missions）

**为什么 P1：** Lurkit 已产品化：「**Performance-based** paid creator campaigns」（Paid Quests）+「Design **missions with specific KPIs** such as views, hours watched, or clicks generated」（Missions & Rewards）。
**方式：`build`**
**建议形态：** 按 KPI（观看量/观看时长/点击）定义任务 → 达标自动核算 → 与交付/ROI 环节打通

---

### 🟠 P1-F3 · 合同 / 支付 —— ✅ **已决策：走 partner**（用户 2026-07-14）

**背景：** 我们原先写的是「明确不做支付/合同/税务」，但**同赛道已把它内置为产品能力**——Lurkit 的支付**确为产品化而非话术**：竞价 → **escrow 资金托管** → **Stripe 结算** → **内容审核通过后放款**（support.lurkit.com 文档证实）。这条边界与市场现实**直接冲突**。

**✅ 决策：`partner`** —— 不自建、也不再是「明确不做」。
- **合同签署** → partner（电子签服务）
- **资金托管 / 结算** → partner（Stripe Connect escrow）
- **KOLMatrix 的边界**：**不碰资金与税务**，只负责 ①触发条件（内容审核通过 / Mission 达标核算）②**人类闸门**（放款需人工确认）③状态可视与追溯（交易记录写入关系资产）

**产品形态（原型已实装）：** 沟通协作（04）环节新增「合同与结算」面板——合同状态 / 托管状态 / 放款条件 / 放款闸门，并显著标注「由 partner 提供」。
**注意：** Lurkit 的合同/谈判是「产品 + 托管服务」**混合形态**（可作 agency 延伸）。我们只做产品 + partner，**边界叙事要讲清楚**：我们管流程与决策，不管资金。

---

### 🟠 P1-F4 · Agent 深度对标（差异化收窄）

**为什么：** 「AI-native / Agent」标签**已不再独占**——GRIN **Gia** 是执行型 Agent（「automatically screens **every** creator」，$1B+ 一方交易数据、180 绩效属性）；HypeAuditor「100% AI-Powered」+ 对话式 HypeAgent。
**行动：** 我们的 Agent 必须在**游戏垂类工作流的深度**上做出差距，而不是比"有没有 AI"。参考 Gia 的"主动筛查每一位"——我们的对标形态可以是**主动扫全库 + 沿六环节主动推进**（原型的"需要你处理"队列已是雏形，可加强）。

## 3. ⚠️ 差异化收窄（必须调整对外叙事）

**验证结论：泛「AI-native / Agent 驱动」叙事已失效**（GRIN Gia / HypeAgent 已在场，~60% 营销人员已用 AI 做发现）。

差异化必须收窄为**三者的组合**：

| # | 差异化 | 证据支撑 |
|---|---|---|
| 1 | **游戏垂类 Agent 工作流**（六环节项目制） | 通用平台没有垂类工作流；垂类对手（Lurkit/Partnier）未做 Agent 化 |
| 2 | **可解释匹配 + 可解释评估** | 竞品 AI 评分多为黑盒；HypeAuditor 95.5% 自报有争议 |
| 3 | **关系资产飞轮** | GRIN 用 $1B 一方交易数据喂 Gia，**证明方向正确**——我们从第一天就为它设计了闭环，但需要时间积累 |

**对落地页 V7 的影响：** 现有落地页强调"智能 Agents"——需要在此基础上把**垂类工作流 + 可解释性**顶到同等位置，否则与 GRIN/HypeAuditor 的叙事同质。

## 5.5 ⭐ 横向借鉴：「有墨公考 AI 业务工作台」（2026-07-15）

分析对象：`docs/product/有墨公考-AI业务工作台.html`（另一工作台类产品原型）。**布局与我们同构**（侧栏+画布+常驻 copilot），说明差异不是审美偏好，是我们缺的东西。

### 已采纳（进 Phase 0 spec r2）

| 借鉴点 | 我们的实现 | spec 位置 |
|---|---|---|
| **角色即产品**（不是权限开关） | 3 角色，各有 access/agent/护栏/数据边界/待办排序口径，**共享一套导航骨架** | D9 · F006 · F008 |
| **否定式护栏**（"我没做什么"比"我很安全"可信） | 每角色 Agent 的 guardrail 全用否定式，注入 system prompt | D13 · F004 |
| **copilotScope 真拒答**（不是摆设） | 越权查询拒答 + 说明边界 + **给替代路径**；切角色清空对话 | D13 · F005 |
| **阈值量化**（"退款超过 ¥1,000"而非"高风险"） | 组合 > $8,000 · 报价 > $2,000 · 批量发信 > 10 人 | D14 · F008 |
| **四级动作分类**（事前规则 ⇄ 事后审计同构） | 部分采纳：闸门归属 + 留痕；完整的动作分类本体留后期 | F008 |

### 已识别但**未**采纳（附理由）

- **`navLabels`（同一实体按角色改名）**：有墨改名是因为业务语义真的不同（"招考项目"↔"内容与活动"）。**我们的 Match 对 BD 与 lead 是同一件事，改名是为改而改。** → spec §7 已记
- **角色专属流水线（workflow-strip）**：有墨用它表达"角色的一天"。**我们驾驶舱已有"项目运转·六环节"+"待办"，再加一层会重蹈"两套并行心智"的覆辙。** 只采纳其 `taskDesc`（待办排序口径按角色不同）——极轻但极聪明。

### 它的坑（**别抄**）

1. **两个自然语言入口行为不一致**（最严重）：dashboard 输入框走"执行走查 modal"，Copilot 输入框走"跳转+解释"。**且更炫的那个更笨**——已验证 `startMockExecution` 拿 `mockExecutionScenarios[state.role]`，**按角色取脚本、完全不看 prompt**。→ 我们只保留一个 NL 入口。
2. **流程只是视图不是实体**：`cls:'warn'` 硬编码，健康度是编的不是算的。← 我们刚被同类问题咬过（假动作 `data-noop`）。
3. **状态机无守卫**：阶段可从"新咨询"直跳"后续办结"，文案讲阶段门、代码一个 guard 没有。← **D10 就是为此而立**。

### 尚可继续借鉴（未进本批）

- ⭐ **AI 执行过程可视化**：六步执行流水线（理解→检索→**校验权限**→生成方案→**等待人工确认**→写入并留痕）+ 执行日志 + "本次数据边界"声明。**闸门是流程的第 5 步，不是外挂弹窗**；确认按钮写进动作数量（"确认执行 3 项动作"）；done 态左键是「撤销」。→ 建议 Phase 1 评估
- **置信度低 → 降级为「待补充」而非显示低分**（"73%"对用户不可行动；翻译成动作把决策成本拿走）→ 建议随知识库（Phase 1）一起做
- **缺口 → 责任人 → 转成工作流对象**（"缺失字段转成顾问跟进问题"是全片最强的一句）→ 我们的知识库缺口目前只到"显性化+上传按钮"，未转待办
