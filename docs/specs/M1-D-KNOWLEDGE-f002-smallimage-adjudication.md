# M1-D-KNOWLEDGE F002 — 小图处置口径开工前审计（快车道即时裁决）

> 快车道豁免条款（§4.6）：Planner 在同会话主上下文即时裁决，裁决段分段标注角色切换。

## 【Generator 提审】矛盾点

同一份 spec 对「图片最短边 ≤10px」给了两个不同处置：

1. **spec §2 F002 行内括注**：「不满足 → parseStatus 直接 failed + parseError 明示，D2 诚实降级不静默」——即**落库存盘、标 failed**；
2. **spec §3 P5**：「上传时校验优于解析时炸」+ **features.json F002 acceptance**：「集成测试：上传落盘落库/路径穿越拒/超限拒/**小图拒**」——「拒」与路径穿越/超限并列，均为 **HTTP 400 拒绝**语义（不落盘不落库）。

两种实现都写得出来，但集成测试断言互斥，必须择一。

**Generator 倾向方案 A（HTTP 400 拒绝）**：与 P5「前置校验优于解析时炸」自洽；≤10px 图片作为素材无信息量，拒收 + 明确报错比入库假 failed 行更诚实；避免垃圾行污染素材列表。

## 【Planner 裁决】——角色切换：主上下文以 Planner 身份裁决

**采方案 A：HTTP 400 拒绝，不落盘不落库。**

理由：
1. §3 P5「上传时校验优于解析时炸」是该约束的**设计动机原文**，§2 行内括注是起草时误把 P6（video 落库 failed）的处置模板串写到了 P5 上——两条 P 的语义本就不同：video 是**合法素材、能力未及**（落库 failed 保留重试语义）；≤10px 图是**无效输入**（上游硬约束 + 无信息量），属输入校验层，同路径穿越/超限一致拒收。
2. features.json acceptance 是 Evaluator 验收执行口径，「小图拒」与两个 HTTP 拒绝并列，维持一致语义可免验收歧义。
3. D2「诚实降级」约束的是**已在系统内的数据**的展示侧，不约束边界输入校验——400 + 明确错误文案同样诚实。

**spec §2 F002 行内括注按本裁决作废**；features.json acceptance 不变。错误文案须明示约束来源（「图片最短边须 >10px（vision 上游约束）」）。
