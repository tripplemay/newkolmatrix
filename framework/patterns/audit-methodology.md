# 审计类批次方法学（框架沉淀）

> **触发条件：** 批次形态为「盘点 / 对照 / 扫描」而非「实现」——前端一致性审计、死代码盘点、安全审计、上线前 audit、PRD 偏差核对。典型是 `executor:evaluator` 的 Evaluator-only 批次，配合 `harness/orchestration-patterns.md` §4/§5 的 fan-out 编排。
>
> **主要读者：** Planner（起草 audit 类 spec 的 acceptance）/ Evaluator（执行）

审计类批次的失败模式与实现类批次不同：实现类批次的风险是「做不出来」，审计类批次的风险是**「做出来一份看着很专业、但大半是噪音的报告」**。§1-§4 针对这个失败模式。

**§5-§6 针对另一个失败模式：报告是真绿的，但当事人高估了它证明了什么**——断言已退化为恒真、探针结构性看不见自己要防的 bug。这两条主要在**复验（reverifying）**阶段用，也适用于任何「拿测试结果当证据」的场合。

---

## 1. 基线词表校准防误报（对照类审计的前置步）

**规律：** 凡是「项目侧相对某个基线是否漂移」的对照类审计（design tokens、组件用法、样式约定、依赖版本），**必须先对基线实物做词表 / 逐字节分类，再扫项目侧**。跳过这一步，所有「项目里出现了 X」都会被当成「项目引入了 X」。

**反面（KOLMatrix FE-AUDIT F003）：** 直接扫项目侧样式，得到一份看似严重的漂移清单。补做模板原件词表后发现 `rounded-2xl` 在**模板自己**的代码里出现 44 次——它是基线的一部分，不是漂移。**约 60+ 处 finding 因此撤回**。

**落地：** audit 类 spec 的 acceptance 应显式包含「基线词表已建立」作为前置 clause，而不是把它留给 evaluator 自觉。

---

## 2. 「组件是否在用」必须走 import 图传递可达性，不能朴素 grep

**规律：** 死代码 / 存活性判定不能用「有没有人 import 它」。必须从**应用真实入口**（`app/` 路由、`main.tsx` 等）出发做**传递可达性**分析——被死代码引用的组件仍然是死的。

**反面（KOLMatrix FE-AUDIT）：** 朴素 grep import 得到 19 个「在用」组件；传递可达性分析实际只有 **12 个**，其余 **7 个是伪存活**——它们只被其他死代码引用，形成孤岛。

**判据：** 从入口可达 = 存活；被引用但整条引用链不接入口 = 死。

### 2.1 风险面按「语义」划，不按「常见目录」划（v1.0.8 — KOLMatrix M1-A-BRIEF 沉淀）

§2 讲存活性别用朴素 grep；这里是对偶问题——**划「哪些文件有某类风险」的审查面时，别按目录经验圈定，要按语义全仓搜**。凭「这类问题一般在 components/ 和 hooks/」圈范围，会漏掉藏在冷门目录里的同类代码。

**反面（KOLMatrix M1-A-BRIEF F002）：** 勘查「拆 NoSSR 后哪些文件有 SSR 风险」时按目录划审查面（components/ + hooks/，spec §1.2 只列了这两处的 6 个文件），漏掉了 `utils/navigation.ts`。而真正让全站每个页面 navbar 标题触发 hydration mismatch 的，恰恰是它里面一行 `if (!isWindowAvailable()) return null` 的服务端/客户端分支——模板遗留代码最爱藏在 utils/ 这类不起眼的地方。

**落地：** 风险面用**语义 grep** 划定，不用目录直觉。例：SSR 风险 = 全仓 `grep -rn 'typeof window\|typeof document\|localStorage\|isWindowAvailable' src`，命中的每个文件都进审查面，无论它在哪个目录。把这条列进 spec 的勘查清单，别留给 Generator 开工时才撞见。

## 3. 汇总层必须是对抗复核层，不是转述层

**规律：** fan-out audit 的汇总 subagent 如果只做「合并 N 份报告」，它就是一个昂贵的复制粘贴——**并行 finder 的共同盲区会原样穿透到最终报告**。汇总 prompt 必须下达**抽查配额**：每份下游报告至少抽查 2 条，**回原件重跑**验证，并显式要求「找出所有 finder 都漏掉的维度」。

**实证（KOLMatrix FE-AUDIT 汇总层实跑）：**
- 推翻 1 处正面分句（finder 说「已收敛」，实物未收敛）
- 修正 1 处推论机制（结论对、归因错）
- **新增 1 条三份并行报告全部漏掉的盲区**：CI 无 DB 导致视觉基线把空区域静默编码为合法（详见 `patterns/web-runtime-patterns.md` §4.3）

最后一条是关键论据：盲区是**并行 finder 结构性看不见**的东西，只有一个被要求「找缺什么」的角色才会发现。

**与 `orchestration-patterns.md` §4 的关系：** §4 的对抗复核只作用于 FAIL/PARTIAL 条目（防误报）；本条作用于**整份报告**（防漏报）。两者方向相反，都要。

## 4. 审计产物脚本化 → 后续批次 acceptance 复跑（审计转回归 harness）

**规律：** audit 批次的交付物**不应止于报告**。凡是可脚本化的检查（token 扫描、组件可达性矩阵、样式词表比对），必须落成 `scripts/test/` 下的可复跑脚本；后续批次的 acceptance 直接引用「跑 X 脚本无回归」。一次性审计因此变成常设回归网。

**实证（两批验证闭环）：**
- FE-AUDIT 沉淀三个扫描脚本（token-scan / component-matrix / style-vocab）
- **ARCH-M05 批末对账时 `token-scan` 抓到真实回归**（53 findings），引出双域 token 收敛（`web-runtime-patterns.md` §5）

没有这层，ARCH-M05 的 token 漂移会静默进生产——它 tsc / lint / 视觉全绿。

**配套纪律：** 复跑脚本适用 `harness/evaluator.md` 的**检测器活性证明**要求——「0 findings」必须先证明脚本还活着（见 `memory/role-context/evaluator.md`）。

## 5. 换实现形态后，既有断言可能静默退化为恒真（复验专用，v1.0.7 — KOLMatrix P2-CLEANUP 沉淀）

**规律：** 修复走了与原实现**不同的形态**时，**不得仅以「原测试由红转绿」作为缺陷消除的证据**。
先做**断言强度审查**：逐条问「这条断言在修复后的实现下，还能区分对错吗？」——尤其子串 / 正则类断言。

**反面（KOLMatrix P2-CLEANUP F003 fix_round1）：** 首轮 harness 的 C3 断言是
`/border-navy-700/.test(className)`。修复把实现从 `isDark ? 'border-navy-700' : 'border-white'`
换成 Tailwind 变体 `dark:border-navy-700` 后，该正则在**浅色态同样为真**（命中变体名内部的子串）。
表面上「harness 由 10 passed/2 failed 转 12 passed/0 failed」，实际**只有一条载荷断言真的转绿**，
C3 那一格已不携带任何信息。实现方据此对外宣称「抓到缺陷的那把尺子原样认可修复」——夸大了证据强度。

**落地：**
- 复验时先跑一遍**反向断言**：把系统置于「本应判红」的状态（如浅色态），确认该断言确实为假
- 断言退化的，补 **discriminating 断言**替代（例：「className 跨深浅切换必须**不变**」+「但 computed 值必须**改变**」，两条合起来才能锁死"零 JS 参与的 CSS 变体"这一形态）
- 结论表述须区分**载荷断言**与**已失效断言**，不得以总数报绿

### 5.1 断言退化的第三种形态：代理判据的前提失效（v1.0.8 — KOLMatrix M1-A-BRIEF 沉淀）

§5 讲的是「换实现形态后断言退化为恒真」。这里是它的同族变种，触发源不同：**测试用一个「代理判据」间接测真实条件，而该代理只在某个架构前提下成立；一旦别的批次改动了那个前提，代理静默失效**——变成恒真或恒假，测试却照常绿。

**反面（KOLMatrix M1-A-BRIEF，同一批两次踩）：** P2-CLEANUP 的 `p2:f002` 用「navbar 主题钮是否存在」代理「React 未 hydrate」，`p2:f004` 用「文案锚点是否出现」代理「数据已到位」。**两个代理都只在全站无 SSR（NoSSR 包裹）时成立**——那时 DOM 里有东西就意味着 hydration 跑完了。M1-A-BRIEF 的 F002 拆掉 NoSSR 恢复 SSR 后，两个代理的前提同时坍塌：元素/锚点现在进了服务端 HTML，早于 hydration 与数据落地。`p2:f002` 那条本是**防假绿的守卫**自己变成恒真；`p2:f004` 则读到中间态间歇性误红。值得记的是：F002 当时修了 `p2:f002`、还把这条规律写进了 proposed-learnings，却**没有把同一条应用到手边的 `p2:f004`**——写下的规律没有横扫到姊妹探针，隔一轮验收才被抓出来。

**落地：**
- **代理判据必须在测试里注释「它成立的前提」**（如「本判据依赖全站无 SSR：元素存在 ⇒ 已 hydrate」）。前提写不出来，说明这个代理本身可疑。
- **改动某架构前提的批次，必须 grep 全部依赖该前提的代理点**——不能只修撞见的那一个。同一失效模式往往成群存在（本例是一对姊妹探针）。
- **换代理判据后，补一条活性证明**：让同一表达式在「相反状态」下翻转（未 hydrate → false、已 hydrate → true）。本批第一版把 hydrate 判据写成 `document.body` 上找 `__reactContainer$`（实际容器是 `document`），恒假——正是被这条活性证明当场抓住。判据换了却不证活性 = 把「可能恒真」换成「可能恒假」，同样是死断言。

## 6. 合成节点探针证明不了「组件真的发出了它」（v1.0.7 — KOLMatrix P2-CLEANUP 沉淀）

**规律：** 验证「组件 X 的样式在条件 Y 下生效」时，用合成节点（自建 `<div class="...">`）只能证明**样式规则存在**，
**不能证明组件真的把这串 class 落到了 DOM**。中间任何一层吞掉 `className`，合成节点探针照样全绿。

**反面（KOLMatrix P2-CLEANUP F003）：** Generator 探针用 `<div id="probe" class="${从源码正则提取}">`
验证 CSS 变体行为——class 非硬编码这点是对的，但不渲染真实组件。而该 feature 的**原始缺陷恰恰就是**
「组件把样式发错了通道、中间层吞掉了它」（`Image` 是纯 div 包装，样式 props 全失效）。
即：这条探针结构性地看不见自己要防的那类 bug。

**落地：** 必须另有一条**挂载真实组件**的路径。零引用组件（产品无路由可达）应建最小挂载 harness——
esbuild 打包真实组件 + link 真实构建产物 CSS + 真实交互路径，而不是退化为合成节点。
参考实现：`scripts/test/f003-harness/`、`scripts/test/f003-reverify/`。
